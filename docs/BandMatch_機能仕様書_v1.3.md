## 1. プロジェクト概要

バンドメンバーを探す個人・グループ向けのマッチングWebサービス「**BandMatch**（仮称）」の機能仕様および開発要件を定義する。既存のテキスト依存型の掲示板・募集サイトからの脱却を狙い、**一覧画面での「顔（写真／アイコン）」と「音源」の即時確認**による、音楽性とスキルの直感的なマッチングをコアバリューとする。

### 1.1 ターゲット
- バンドメンバーを探している個人（加入希望者）
- 新規メンバー／サポートメンバーを探している既存バンド（グループ）
- 一緒に練習する相手を探している個人

### 1.2 コアバリュー
詳細ページへ遷移せずとも、検索結果の一覧の段階で「どんな人（顔）か」「どんな音（音源）か」を即座に確認できること。判断材料を一覧に集約し、画面遷移を最小化することで、直感的で高速なマッチング体験を提供する。

### 1.3 収益モデル
- **フリーミアム課金（C2C）:** 基本機能は無料。上位表示・高度な検索フィルタ・グループチャット等のプレミアム機能を課金対象とする。
- **送客手数料・広告（B2B2C）:** 音楽スタジオの検索・予約連携によるアフィリエイト報酬、およびライブハウス・スタジオ等の広告出稿（検索一覧内のネイティブ広告枠）。

## 2. 開発方針・技術構成

### 2.1 開発体制と方針
開発者自身がノーコード／ローコードツールを用いて構築する。マッチング体験（音源・検索）の質を優先し、まずレスポンシブWebでローンチ。将来的にネイティブアプリ（iOS／Android）化を計画している。

### 2.2 技術スタックと選定理由
将来のネイティブアプリ化を見据え、Webとアプリを同一プロジェクトから出力でき、かつコード書き出しによる拡張余地を残せる **FlutterFlow** を主軸に採用する。バックエンドは、多条件の絞り込み検索（本サービスの心臓部）に構造的に強いSQLベースの **Supabase** を採用する。

| レイヤー | 採用技術 | 役割・選定理由 |
|---|---|---|
| フロント／ロジック | FlutterFlow | 画面・画面遷移・ワークフロー。Web出力でローンチし、同一プロジェクトからネイティブアプリを出力可能。Flutterコード書き出しで将来の拡張・内製化にも対応。 |
| バックエンド／DB | Supabase | PostgreSQL・認証・ファイル保存（Storage）・リアルタイム通信。複合条件検索にSQLが有利。FlutterFlow公式連携あり。 |
| 決済 | Stripe | プレミアム課金のサブスクリプション決済（Phase 2 で有効化）。FlutterFlow標準連携。 |
| 音源・動画 | YouTube / SoundCloud 埋め込み（Phase 1）＋ Supabase Storage 直接アップロード（Phase 2） | ポートフォリオの試聴・視聴。 |
| スタジオ連携 | 外部予約サイトへのアフィリエイトリンク（Phase 2）→ API直接予約（Phase 3） | マネタイズ導線。 |

> **将来のネイティブアプリ化に関する重要事項:** Bubble等のWeb特化型ノーコードではアプリ化時に二重開発が発生する。FlutterFlowは同一プロジェクトからWeb／iOS／Androidを出力でき二重開発を回避できる。トレードオフとして学習曲線がやや急である点に留意する。

### 2.3 動作環境

| 項目 | 内容 |
|---|---|
| 提供形態（Phase 1） | レスポンシブWeb（PCブラウザ／スマートフォンブラウザ） |
| 対応ブラウザ | 各最新版の Google Chrome / Safari / Microsoft Edge / Firefox |
| 将来の提供形態 | 同一FlutterFlowプロジェクトから iOS / Android ネイティブアプリを出力（Phase 3） |
| 提供エリア（Phase 1） | 関東圏に限定（第5章 集客戦略を参照） |
| 言語・タイムゾーン | 日本語のみ／JST固定（多言語は将来） |

## 3. フェーズ設計

限られた期間でのローンチと段階的成長を両立するため、機能を4フェーズに分割する。**Phase 1（MVP）はコア体験に徹底して絞り込み**、早期ローンチと市場検証を優先する。

- **Phase 0 ― 事前登録（ウェイトリスト）:** メールアドレスのみを収集する事前登録ページを先行公開。SNSでの開発過程発信（ビルド・イン・パブリック）と連動し、ローンチ時の一斉告知に備える見込みユーザーの受け皿。
- **Phase 1 ― MVP（最初のローンチ・関東圏限定）:** 「探す・つながる・やり取りする」というコア体験に、信頼性を担保するレビューと最低限の安全対策（通報・ブロック）、招待機能、最低限の運営管理機能を加えた最小構成。
- **Phase 2 ― 収益化・全国展開:** プレミアム課金、スタジオ予約連携（アフィリエイト）、グループチャット・日程調整・ファイル送付等の有料機能、提供エリアの全国展開。
- **Phase 3 ― 拡張:** 評価フィルタ、人気ランキング、求人情報、スタジオAPI直接予約、ネイティブアプリ化。

> **Phase 2以降の機能の見せ方（UI方針）:** 主要機能（プレミアム・スタジオ予約・グループチャット・日程調整・ファイル送付）のみを、UI上に **グレーアウト＋「近日公開」ラベル** で先行表示する。全機能は並べない（未完成な印象を避ける）。

## 5. 集客・ローンチ戦略

マッチングサービスは「相手がいなければ使えない」鶏卵問題を宿命的に抱える。初期に「人がいる状態」を作るため、以下3つを組み合わせる。

- **5.1 関東圏へのエリア限定（密度の確保）:** バンドは対面で音を合わせるため地理的密度が体験の質に直結する。Phase 1は関東圏に限定し、Phase 2で全国展開する。
- **5.2 既存コミュニティとの連携:** 音楽スタジオ・軽音サークル・楽器店等に協力を仰ぎ初期ユーザーを流し込む。登録時に「どこで知ったか」を任意取得し流入経路を計測する。
- **5.3 SNSでの開発過程発信（ビルド・イン・パブリック）:** 見込みユーザーに届く発信を継続し、ローンチ時点で「待っている人がいる」状態を作る。関心の受け皿を Phase 0 の事前登録ページとする。

## 6. アカウント・認証・本人確認

### 6.1 認証・連絡先
- **メールアドレス:** 認証・連絡に必須・一意。
- **電話番号:** **必須**とし、**SMSによる本人確認**を行う。狙いはなりすまし・複数アカウント・業者流入の入口抑止。登録ハードルは上がるが、マッチングサービスとしての安全・信頼を優先する判断。
- **SNS連携:** Google / Apple。

### 6.2 アカウント状態と退会（論理削除）
- アカウント状態を **active / suspended（凍結）/ withdrawn（退会）** で管理。
- **退会は論理削除**：本人は利用不可となるが、相手側の会話履歴・レビューには「退会したユーザー」として残す（履歴の一貫性を保つ）。
- 凍結・退会ユーザーは検索・一覧に出現しない。

### 6.3 登録フロー
`メール／SNSでアカウント作成` → `電話番号SMS本人確認` → `アプリ利用目的の選択` → `プロフィール入力（必須項目）` → `規約同意（記録）` → `検索一覧へ`。

## 7. プロフィール構造（v1.3の中心）

プロフィールを **4ブロック** に再構成する。募集条件は個人・バンド共通で、「メンバー募集中」をオンにした人だけが入力・表示する。

### ブロックA：アプリ利用目的（登録必須・検索軸）
以下から選択（複数選択可）。検索の主要軸にもなり、「加入したい人」と「募集しているバンド」が出会いやすくなる。
- バンドメンバーを募集したい
- バンドに加入したい
- 一緒に練習する相手を探したい

### ブロックB：自分の属性

| 項目 | 必須 | 選択肢・備考 |
|---|:--:|---|
| 担当パート（複数選択可） | ○ | ボーカル／ギター／ベース／ピアノ・キーボード／ドラム／パーカッション／管楽器／弦楽器／作詞作曲・アレンジャー／DJ／ダンサー／その他（自由記述）＝**12択** |
| 活動歴 | ○ | はじめたばかり／初心者／経験者／プロ志向 |
| 活動頻度 | 任意 | 月1〜2回／週1〜2回／毎日 |
| 活動日 | 任意 | 曜日・時間帯 |
| 楽器演奏の有無 | 任意 | 楽器をやっている／やっていないが音楽好き |
| 社会人／学生 | 任意 | 属性タグ。検索フィルタ対象（v1.0から継続） |
| 好きなアーティスト | 任意 | 自由記述・**検索非対象** |
| 所有機材 | 任意 | 自由記述 |
| 自己紹介 | 任意 | 0–1000字 |

### ブロックC：目指すスタイル

| 項目 | 選択肢 |
|---|---|
| コピー／オリジナル志向 | コピーバンド中心／オリジナル曲中心 |
| 音楽ジャンル（複数選択可） | ポップス／ロック／ハードロック・ヘビーメタル／パンク・メロコア／ハードコア／スラッシュメタル・デスメタル／ビジュアル系／ファンク・ブルース／ジャズ・フュージョン／カントリー・フォーク／スカ・ロカビリー／ソウル・R&B／ゴスペル・アカペラ／ボサノバ・ラテン／クラシック／ヒップホップ・レゲエ／ハウス・テクノ／アニソン・ボカロ＝**18択** |
| イメージする楽器構成（複数選択可） | パート12択と同じ |

### ブロックD：募集条件（「メンバー募集中」オン時のみ入力・表示。個人・バンド共通）

| 項目 | 選択肢 |
|---|---|
| メンバー募集中 | on/off（一覧に「募集中」タグ表示） |
| 募集するパート（複数） | パート12択 |
| 募集する性別 | 指定なし／男性／女性 |
| 募集する年齢 | 範囲（下限・上限） |
| 募集目的 | 軽くセッション／曲作り／バンド加入者募集 |

### 7.1 登録必須項目（最小限）
登録ハードルを下げるため必須は最小限とし、他は任意で後から追加できる。

- **必須:** アプリ利用目的、担当パート、活動歴、活動エリア、音楽ジャンル、表示名、生年月日
- **必須ステップ（認証）:** 電話番号のSMS本人確認（プロフィール項目とは別に、登録フロー上の必須ステップ）
- **任意:** 上記以外のすべて（募集条件は「募集中」オン時のみ入力）

> **設計変更点:** 旧「目的の真剣度（seriousness）」は **廃止し、活動歴（はじめたばかり〜プロ志向）に統合** した。

### 7.2 メディア（写真・音源）
- プロフィールのアイコン画像は Phase 1 から Supabase Storage に保存。未設定時はイニシャル＋自動配色のプレースホルダを表示。
- 音源・動画は Phase 1 は **YouTube / SoundCloud のURL埋め込みのみ**。直接アップロード（Storage）は Phase 2。著作権リスク回避のためこの順序とする。

## 8. グループ（バンド）

- グループ（バンド）は独立したアカウントではなく、**複数の個人ユーザーが集まって形成する箱**として設計する（独立テーブル＋中間テーブル）。
- **権限は admin / member の2段階。** 会話は「バンドとして」行う（表示はバンド名、操作は管理者、履歴はバンドに帰属）。
- **管理者退会時:** 最古参メンバー（参加が最も古い member）へ自動昇格。member が一人もいなければバンドを凍結。
- バンドの掛け持ち上限なし・同名バンド許可（IDで区別）。
- バンドにも「目指すスタイル（C）」「募集条件（D）」を個人と同じ項目で付与する。

---

## 9. マッチング（メッセージリクエスト）

- 旧「いいね／スカウト」の2ボタンは **「メッセージリクエスト」に一本化**（メッセージを送る）。
- **片側リクエスト＋承認制:** 送信者がリクエストを送り、受信者が承認して初めてメッセージルームが開設される。承認前の一方的なやり取りは発生しない。
- **付属メモ添付可**（0–300字）。承認後にルーム冒頭へ表示。
- 拒否後の再送は不可。両想い（相互リクエスト）は自動成立。有効期限なし。
- **送信制約:** 自分自身・ブロック相手・退会/凍結ユーザーにはリクエストを送れない。

---

## 10. メッセージ

- 承認後にルーム開設。**通数無制限**。
- Phase 1 は **text / stamp**（スタンプは無料・基本セット15種程度）。メッセージ本文は1–2000字。
- 参加者は「種別（個人／バンド）＋ID」で汎用保持し、将来のグループチャット拡張に対応する。
- **既読は Phase 1 は記録のみ**（表示は Phase 2 の有料機能）。

---

## 11. エリア（地域階層）

- **都県／市区／駅の3階層。** 関東の駅は網羅インポートする。
- 複数駅を登録可。検索は階層で行う（都県→市区→駅）。
- **非公開にできるのは市区と駅**（都県は公開固定）。非公開項目はその条件で検索ヒットしない。

## 12. 検索・並び順

### 12.1 検索フィルタ（Phase 1）
- **利用目的**（募集したい／加入したい／練習したい）＝検索の主要軸
- 担当パート（12択）、活動エリア（都県/市区/駅の階層）、年齢、性別、**音楽ジャンル（18択）**
- **活動歴**（旧・真剣度を置換）、活動頻度（任意）、社会人／学生
- **「メンバー募集中のみ」トグル**
- 「音源・動画登録ありのみ」トグル、「最終ログイン日時」によるアクティブ絞り込み
- 好きなアーティストは検索対象外。非公開項目（年齢・性別・市区・駅）はヒットしない。
- 評価（レビュー）による絞り込みは Phase 3（UI上はグレーアウトで先行表示）。

### 12.2 並び順
一覧はスコアの高い順に並べる（詳細な計算式はプログラム仕様書 v1.3）。距離はエリア階層で簡易評価、評価はコールドスタート配慮、重みは運用調整。プレミアム上位表示は Phase 1 では無効（1.0）。無限スクロールで20件ずつ読み込む。

### 12.3 インライン音源再生（最重要）
カード上に再生ボタンを配置し、詳細ページに遷移せずその場でサンプル音源を試聴できる。一覧内の再生は1曲のみ（別カード再生で前を停止）。

---

## 13. レビュー

- 対象：**個人にもバンドにも**評価が付く。形式は星5段階＋コメント。
- **記入可能なのはマッチ成立後（承認され、やり取りが発生した相手）のみ。**
- **相互ブラインド:** 双方が投稿し終えるまで相手の評価は非表示。両者投稿後にまとめて公開。狙いは報復レビュー・忖度評価の防止。
- **14日ルール:** 片方のみ投稿の場合でも、投稿日時から14日経過で片側だけでも公開する。

---

## 14. 安全対策・運営

- **通報:** 不適切なユーザー・メッセージを運営へ報告（ユーザー単位）。**自動制限は行わず運営の手動判断**でアカウント状態を変更する。
- **ブロック:** サイレント化（相手に明示サインを出さず、検索・一覧からも消える）。
- **運営向け管理機能（Phase 1 に最低限実装／ユーザーアプリとは別）:** 通報確認・対応状況更新、ユーザー凍結／解除、強制退会（論理削除）、監査ログ。
- 登録時に生年月日を取得し、利用規約・プライバシーポリシーへの同意を必須とする（本格的な年齢確認は Phase 2）。
- 投稿規約に基づく基本的な禁止ワードフィルタを設ける（運用で拡充）。

---

## 15. 招待機能

ユーザー数がサービスの生命線であるため、招待を Phase 1 の **必須機能** とする。

- **友達招待:** 招待リンク／コードを発行し、SNS・メール等で共有。経由での登録を追跡する。
- **バンド内招待:** バンド管理者が既存メンバーをアプリへ招く。承諾するとバンドのメンバーに追加。「個人が集まってバンドを形成する」設計上の必須導線。
- **特典・追跡:** Phase 1 は特典なし。Phase 2 で課金と連動し招待者・被招待者へプレミアム付与等を検討。誰の招待で何人が登録したかを集計できる。

---

## 16. フリーミアム設計（無料／有料の線引き）

集客最優先のため、**「マッチングが成立するまでの体験」は無料で開放し**、課金は「成立後、もっと便利に・活発に・有利に使いたくなる箇所」に置く。

- **無料（探す・つながる・やり取りする）:** プロフィール作成・写真/アイコン設定、検索・一覧閲覧・音源試聴、メッセージリクエスト送信（枠制限なし）、承認相手とのメッセージ（通数無制限）、レビューの投稿・閲覧、招待。
- **有料（Phase 2）:** 検索結果の上位表示、高度な検索フィルタ、足あと／既読の確認、グループチャット、日程調整、ファイル送付（画像・動画・音声）。

> **通数・枠制限の方針:** メッセージ通数制限もリクエスト（旧スカウト）枠上限も Phase 1 では設けない（集客優先）。乱発が問題化した段階（Phase 2）で上限導入とその解放を課金ポイントとして実装する。制限は後から追加できるが、最初からきつくすると人が集まらない、が基本思想。

---

## 17. 収益・広告

- **検索一覧のネイティブ広告:** カード間に、数件ごとに広告枠を差し込む（ライブハウス・スタジオ等の出稿。**「PR」表記**）。エリアターゲティング・優先度を持つ。
- **スタジオ予約連携:** Phase 2（アフィリエイト送客）→ Phase 3（API直接予約）。

---

## 18. 規約・同意の管理（コンプライアンス）

- 利用規約／プライバシーポリシー／投稿規約／免責事項の **本文をバージョン管理** する。
- ユーザーが **いつ・どの版に同意したか** を記録する。規約改定時は再同意を求める。法務・トラブル対応の根拠とする。

---

## 19. UI・ブランド・デスクトップ対応

- **デザインテイスト:** Z世代向けの **Bumble風**（イエロー基調 #FFC629、大きな写真カード、丸型UI、余白多め、ポップ）。モバイル版はワイヤーフレーム v3 で確立済み。
- **ロゴ:** 検索一覧の左上にロゴ「BandMatch」を配置（Phase 1 は仮のワードマークでよい）。
- **レスポンシブ:** モバイルは1カラム＋ボトムタブ。**デスクトップ（1280px想定）は回遊型レイアウト**＝上部に横並びグローバルナビ、左にフィルタサイドバー、中央にカードグリッド、右にプロフィール／広告カラム。FlutterFlowのブレークポイントで出し分ける。
- **Phase 2予告:** 主要機能のみグレーアウト＋「近日公開」ラベルで表示。

## 20. 非機能要件

| 項目 | 要件・方針 |
|---|---|
| 安全性 | 通報・ブロック・SMS本人確認（Phase 1）。年齢確認の強化（Phase 2）。 |
| 個人情報保護 | プライバシーポリシー整備。Supabase の Row Level Security で本人以外がデータを参照できないようにする。 |
| 著作権 | 音源は Phase 1 では YouTube/SoundCloud の正規埋め込みのみ。Phase 2 の直接アップロード導入時に投稿規約で権利範囲を明確化。 |
| パフォーマンス | 一覧のインライン再生・複合検索が主要負荷。インデックス設計とページング（無限スクロール20件）で応答性を確保。 |
| コスト | FlutterFlow（月額約30ドル〜）、Supabase（無料枠〜従量）、Stripe（決済手数料）、SMS送信コスト。従量課金の増加を運用計画で監視。 |
| 拡張性 | ネイティブアプリ化・全国展開を見据え、エリアはマスタ管理、機能はフェーズ分離で疎結合に保つ。 |
| 言語・TZ | 日本語のみ・JST固定。 |

## 21. 確定した主要な意思決定（サマリ）

1. **技術:** FlutterFlow ＋ Supabase ＋ Stripe。将来のアプリ化を見据え二重開発を回避。
2. **範囲:** Phase 1 はコア＋レビュー＋通報/ブロック＋招待＋最低限の運営管理。関東圏でローンチ。
3. **プロフィール4ブロック再構成:** 利用目的（必須・検索軸）／自分の属性／目指すスタイル／募集条件（募集オン時のみ・個人/バンド共通）。
4. **選択肢刷新:** 担当パート12択、音楽ジャンル18択。真剣度は廃止し活動歴に統合。社会人/学生は任意項目として継続。
5. **登録必須は最小限:** 利用目的・パート・活動歴・エリア・ジャンル・表示名・生年月日（＋SMS認証ステップ）。
6. **マッチング:** メッセージリクエストに一本化。片側リクエスト＋承認制、メモ添付可、拒否後再送不可、両想い自動成立。
7. **メッセージ:** 承認後ルーム、通数無制限、text/stamp、既読はP1記録のみ。
8. **エリア:** 都県/市区/駅の3階層。市区・駅は非公開可、都県は公開固定。
9. **レビュー:** 星5＋コメント、成立後のみ、相互ブラインド＋14日ルール。
10. **収益:** 検索一覧のPR広告枠を Phase 1 で実装。プレミアム・スタジオ連携は Phase 2 以降。
11. **UI:** Bumble風・左上ロゴ・デスクトップ回遊型レイアウトを定義。Phase 2機能はグレーアウト＋近日公開。
